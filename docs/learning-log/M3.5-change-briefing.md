# M3.5 — Change Briefing

The feature that briefs the next agent on what the previous one changed. Storage
and capture are built bottom-up: M3.5a is the persistence + diff foundation only;
acquire-time snapshotting (M3.5b) and release-time recording (M3.5c) wire it up later.

---

## M3.5a — change-log storage + diff foundation (2026-06-26)

**Built:** The persistence floor for change briefing, no tool wiring. Three pieces:
(1) migration 005 adds `locks.content_snapshot` (the per-lock baseline, captured at
acquire in M3.5b) as an IN-PLACE nullable column, and creates the append-only
`change_log` table (the recorded change, created at release in M3.5c, which must
OUTLIVE the lock). (2) `core/diff.ts` — `diffContent(before, after)` produces a
git-format unified diff via `git diff --no-index`. (3) `core/changes.ts` —
`recordChange` / `getChanges`, pure repo-scoped storage. 78 tests green (+16).

**The central design idea — opposite lifecycles, opposite homes:**
- The SNAPSHOT (baseline content at acquire) is useful only until the diff is computed;
  it dies WITH the lock → it lives on the `locks` row (`content_snapshot`).
- The CHANGE RECORD must brief the NEXT acquirer, who arrives AFTER this lock is gone;
  it must OUTLIVE the lock → its own `change_log` table.
  Same data-flow, but co-locating them would either leak the snapshot past its use or
  kill the record before anyone reads it.

**Why content_snapshot is nullable-with-no-default:** the engine's INSERT does not list
it yet (M3.5b), so today every row arrives without one — a legitimately absent snapshot
must be legal. This is the deliberate INVERSE of S1a's repo_root (a non-null identity
column where a missing value is a bug worth failing loud over). Missing identity = bug;
missing baseline = normal pre-capture state.

**Why diff is NOT NULL but summary/diff_stat are nullable:** the diff is the FLOOR —
mechanical, always computable, always stored (even "" for a no-op). The summary is
ENRICHMENT — an agent may write it at release. MeshLock must brief fine on a diff alone,
so the prose fields are nullable and read back as null without complaint.

**Why change_log is a log, not state:** `locks` keeps ONE row per (repo_root, path,
branch) — current truth, hence UNIQUE on the triple. `change_log` keeps MANY rows per
path over time — that accumulated history IS the briefing. So a surrogate INTEGER id and
deliberately NO UNIQUE on the identity triple.

**The fiddly bit — git diff --no-index exit codes:** git exits 0 when files are IDENTICAL
and 1 when they DIFFER. For us "they differ" is the successful, normal case. So diff.ts
uses spawnSync (reports `.status`) not execFileSync (throws on any non-zero): 0 → "",
1 → stdout, anything else → throw. git has no "diff two strings" mode (--no-index takes
two PATHS), so we write both sides to a scratch temp dir, diff the files, delete the dir.
Synchronous throughout, to match the synchronous lock engine.

**Storage is dumb on purpose:** recordChange stores whatever it is handed, unconditionally.
Whether a change is worth recording (skip empty diffs?) is the caller's policy in M3.5c,
not storage's. Decision out of storage = policy can change with no migration.

**In-place ALTER vs table rebuild:** 002/003/004 rebuilt `locks` (new table → copy → drop
→ rename) because they changed a multi-column UNIQUE, which SQLite cannot alter in place.
005 just adds a plain nullable column — a cheap metadata change, so a one-line
`ALTER TABLE ... ADD COLUMN`. Fingerprint in the schema dump: content_snapshot is jammed
onto the end of the `branch` line rather than getting its own tidy line — bolted on, not
rewritten in a rebuild.

**Concepts:** opposite-lifecycle reasoning (state row vs append-only log); nullable-when-
optional vs non-null-when-identity (this vs S1a, mirror images); the floor/enrichment split
(diff NOT NULL, summary nullable); git diff --no-index exit-code-1-means-success and
spawnSync-vs-execFileSync; the optional `?` erases at runtime so storage must `?? null`
(undefined would crash better-sqlite3); snake_case row ↔ camelCase record mapping at the
module boundary; null vs undefined as distinct values driving a three-way branch filter;
storage-dumb / policy-in-caller separation.

**Interview Qs:**

Q: content_snapshot is nullable-with-no-default, but repo_root (S1a) is non-null. Both
describe "a value that might not be supplied yet." Why does one fail loud and the other stay
quietly null?
A: one fails because no root is not supposed to happen whereas no diff is the starting point
so it can be null. (Right reasoning — identity (repo_root) absent = a bug, fail loud;
baseline absent = the legitimate starting state, allow null. One slip: the nullable column
is content_snapshot, the SNAPSHOT, not the diff. The diff in change_log is the opposite —
NOT NULL, the always-present floor.)

Q: recordChange does `record.summary ?? null`. What breaks if you delete the `?? null` and
pass record.summary straight to better-sqlite3 when the caller omitted summary? Why doesn't
the `summary?: string | null` type alone prevent it?
A: I'm not sure. (better-sqlite3 throws on a bound `undefined`. An omitted optional field IS
`undefined` at runtime, because the `?` marker is compile-time only and ERASES after
compilation — so the type gives zero runtime protection. `?? null` converts that undefined
into a real SQL NULL, which SQLite accepts; a real string passes through untouched.)

Q: diff.ts uses spawnSync instead of execFileSync. Given git exits 1 when files differ, what
exactly would go wrong with execFileSync, and which existing test would catch the regression?
A: because it throws on any non zero exit, 1 meaning differences found which is the correct
output. (Correct. execFileSync throws on exit 1, so every REAL diff — the normal case —
would throw and likely be swallowed by M3.5c's graceful degradation, silently blanking
briefings. The tripwire is diff.test.ts "does not throw when the inputs differ", red at
build time.)

Q: change_log has a surrogate id and NO UNIQUE on (repo_root, path, branch), while locks has
the opposite. Explain the one-word reason in terms of what each table IS (state vs … ?).
A: because locks get deleted and the changelog has to survive the deletion. (That's the
lifecycle reason for the SEPARATE TABLE — correct, but a different facet. The surrogate-id /
no-UNIQUE reason: locks is STATE (one row per identity = current truth → UNIQUE on the
triple); change_log is a LOG/HISTORY (many rows per identity over time = the whole point →
surrogate key, no UNIQUE).)

Q: getChanges treats branch: undefined, branch: "main", and branch: null as three different
queries. What does each return, and which TS feature makes the three distinguishable?
A: for undefined it builds the branch clause conditionally so an omitted branch means any,
for null and string it handles both of them safely. (Mechanism right. The three results:
undefined → ALL branches' changes; "main" → only main; null → only branchless. The enabling
TS feature: null and undefined are DISTINCT values in TS, so `query.branch === undefined`
forks "omitted" away from "explicitly null" — the latter then hits `branch IS ?` null-safely.)

**Still fuzzy:**

- Why `?? null` is needed: the optional `?` erases at runtime, so an omitted field is
  `undefined`, which better-sqlite3 rejects — the type can't guard a runtime value (Q2).
- The surrogate-id / no-UNIQUE reason for change_log: it's a LOG (many rows per identity =
  history) vs locks as STATE (one row per identity); separate from the survival/lifecycle
  reason for it being its own table (Q4).
- That null and undefined are distinct values in TS, which is what lets the branch filter
  fork three ways on one optional field (Q5).
